## Переписанный вариант

Если вы следите за AI-агентами, вы постоянно слышите «MCP». Давайте разберёмся, что это такое и почему это важно.
## Что такое MCP
**MCP (Model Context Protocol)** — это открытый протокол, который позволяет AI-агентам подключаться к внешним инструментам и источникам данных. Придуман Anthropic в конце 2024 года, сейчас поддерживается всем сообществом.
Простая аналогия: если AI-агент — это человек за компьютером, то MCP — это USB-порт. Через него можно подключить что угодно: файловую систему, базу данных, GitHub, API сервисов, календарь, почту. Без MCP каждый инструмент требовал бы отдельной интеграции, написанной вручную.
## Как это работает
Архитектура MCP — клиент-серверная:
```
AI-агент (клиент MCP)
    │
    ├── MCP-сервер GitHub ─── поиск issues, создание PR
    ├── MCP-сервер Файлы ──── чтение, запись, поиск по папкам
    ├── MCP-сервер База данных ── SQL-запросы
    └── MCP-сервер Почта ──── отправка, чтение писем
```
Агент (клиент) подключается к MCP-серверам при запуске, узнаёт какие инструменты они предоставляют, и дальше может вызывать их так же, как свои встроенные функции.
## Что такое MCP-сервер
**MCP-сервер** — это программа, которая предоставляет конкретный набор инструментов через стандартизированный интерфейс.
Например, GitHub MCP-сервер даёт агенту инструменты: `list_issues`, `create_pull_request`, `search_code`. Агент не знает как устроен GitHub API внутри — он просто вызывает `mcp_github_list_issues()` и получает результат в стандартном формате.
Сервер может быть:
- **Локальным** (stdio) — запускается как подпроцесс на той же машине. Примеры: `npx @modelcontextprotocol/server-github`, `uvx mcp-server-time`
- **Удалённым** (HTTP) — работает на сервере, доступен по URL. Пример: `https://mcp.mycompany.com/v1/mcp`
## Два транспорта MCP
| Транспорт | Как работает | Когда использовать |
|-----------|-------------|-------------------|
| **stdio** | Агент запускает сервер как дочерний процесс, общение через stdin/stdout | Локальные инструменты: файлы, git, утилиты |
| **Streamable HTTP** | Агент подключается к серверу по HTTP, общение через POST-запросы | Удалённые сервисы, командные серверы, продакшен |
Один агент может одновременно использовать оба типа — часть серверов локальные, часть удалённые.
## Безопасность
MCP спроектирован с учётом безопасности:
- **Изолированное окружение.** Для stdio-серверов агент НЕ передаёт все переменные окружения (API-ключи, токены) дочернему процессу. Только безопасный минимум: `PATH`, `HOME`, `USER`.
- **Явные разрешения.** Если серверу нужен API-ключ — вы указываете его в конфиге явно: `env: { GITHUB_TOKEN: "..." }`.
- **Авто-цензура ошибок.** Если MCP-сервер возвращает ошибку с токеном в тексте — агент вырезает его перед тем как показать модели.
## MCP vs API — в чём разница
| | Прямой API | MCP |
|---|---|---|
| Интеграция | Пишешь код для каждого сервиса | Универсальный протокол, один раз |
| Обнаружение инструментов | Читаешь документацию, пишешь обёртки | Автоматическое: `list_tools()` → готово |
| Переиспользование | Каждый проект с нуля | Один MCP-сервер — все агенты |
| Стандартизация | У каждого API свой формат | Единый JSON-формат ответов |
MCP не заменяет API — он стандартизирует способ их подключения к агентам.
## Где уже используется
- **Claude Desktop** — нативная поддержка MCP с ноября 2024
- **OpenCode** — MCP для подключения любых инструментов
- **Hermes Agent** — встроенный MCP-клиент с автообнаружением
- **Cursor, Windsurf** — поддержка MCP в IDE
- **Сотни community-серверов** — от Google Maps до PostgreSQL
## Как выглядит на практике
В конфиге агента это одна секция:
```yaml
mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "ghp_..."
  time:
    command: "uvx"
    args: ["mcp-server-time"]
  company_api:
    url: "https://mcp.internal.company.com/mcp"
    headers:
      Authorization: "Bearer sk-..."
```
Всё. Агент при старте подключится ко всем трём серверам, найдёт их инструменты и зарегистрирует как `mcp_github_list_issues`, `mcp_time_get_current_time`, `mcp_company_api_*`.
## Зачем это нужно вам
MCP решает проблему фрагментации. Раньше для каждого AI-проекта нужно было писать интеграции под каждый сервис: GitHub, Jira, Slack, базы данных. Теперь — один раз поднимаете MCP-сервер, и все агенты в экосистеме могут его использовать.
Это как HTTP для веба, только для инструментов AI-агентов.
