# Что такое ACP — протокол, на котором агенты общаются друг с другом

> [!quote]- Оригинал (голосовая заметка)
> *(нет голосового оригинала)*

## Переписанный вариант

Если MCP — это язык для общения агента с инструментами, то ACP — это язык для общения агентов друг с другом.
## Что такое ACP
**ACP (Agent Communication Protocol)** — открытый протокол для взаимодействия между AI-агентами. Разработан как открытый стандарт под Linux Foundation вместе с BeeAI (его референсной реализацией). С июня 2025 года ACP является частью протокола **A2A (Agent-to-Agent)** под эгидой Linux Foundation.
Простая аналогия: если агенты — это люди в офисе, то ACP — это корпоративный мессенджер. Через него они ставят друг другу задачи, передают результаты, просят помощи.
## Зачем нужен ACP
Современные AI-системы страдают от фрагментации:
- **Разные фреймворки.** LangChain, CrewAI, BeeAI, кастомные решения — все говорят на разных языках
- **Разные команды.** Одна команда делает поискового агента, другая — агента для генерации отчётов
- **Разная инфраструктура.** Часть агентов в облаке, часть локально
- **Нет общего интерфейса.** Невозможно просто «попросить» чужого агента что-то сделать
ACP решает эти проблемы, предоставляя единый стандарт коммуникации, который работает поверх любого стека.
## Как работает ACP
ACP построен на простом RESTful API с поддержкой стриминга:
```
Агент-оркестратор
    │
    ├── POST /runs  ──────────►  Агент-ресёрчер
    │   {"task": "найди статьи..."}   │
    │◄── результат ◄─────────────     │
    │                                 │
    ├── POST /runs  ──────────►  Агент-писатель
    │   {"task": "напиши черновик..."}
    │◄── стрим ◄─────────────────
    │
    └── POST /runs  ──────────►  Агент-редактор
        {"task": "вычитай..."}
```
## Ключевые возможности ACP
**Стриминг.** Агенты могут отправлять результаты по мере готовности — не нужно ждать полного завершения задачи.
**Stateful-сессии.** Агенты запоминают контекст взаимодействия. Можно отправить запрос, получить ответ, уточнить — и агент помнит историю.
**Agent Discovery.** Агенты могут «представиться» и рассказать что они умеют. Оркестратор автоматически узнаёт какие агенты доступны и какие задачи могут выполнять.
**Agent Manifest.** Каждый агент описывает себя в манифесте: какие задачи решает, какие модели использует, какие инструменты доступны.
**Распределённые сессии.** Один разговор может пережить перезапуск агента — сессии персистентны.
**Артефакты.** Агенты могут генерировать файлы, отчёты, дашборды — и возвращать их как часть ответа.
**Await External Response.** Агент может приостановить работу в ожидании ответа от человека — и продолжить, когда ответ получен.
**High Availability.** Встроенная поддержка HA для продакшен-нагрузок.
## ACP vs MCP — в чём разница
| | MCP | ACP |
|---|---|---|
| **Для чего** | Агент ↔ Инструменты | Агент ↔ Агент |
| **Направление** | Агент вызывает инструмент | Агенты общаются двусторонне |
| **Протокол** | stdio / HTTP | RESTful API |
| **Стриминг** | Нет (запрос-ответ) | ✅ Полный стриминг |
| **Сессии** | Нет | ✅ Stateful + персистентные |
| **Discovery** | `list_tools()` | Agent Manifest + Discovery |
| **Человек в цикле** | Нет | ✅ Await External Response |
| **Под чьей эгидой** | Anthropic → сообщество | Linux Foundation |
Проще говоря: **MCP даёт агенту руки** (инструменты), **ACP даёт агентам коллег** (других агентов).
Они комплементарны. В идеальной архитектуре MCP-серверы предоставляют инструменты, а ACP позволяет агентам передавать друг другу работу.
## Где уже используется
- **BeeAI** — референсная реализация ACP от IBM
- **OpenCode** — поддержка ACP для связи между агентами
- **BeeAI Platform** — платформа для тестирования, деплоя и шаринга ACP-агентов
- **MCP Adapter** — мост между MCP и ACP: MCP-сервер может притвориться ACP-агентом
## Кусочек кода для понимания
Отправить задачу агенту — простой HTTP-запрос:
```bash
curl -X POST https://agent.example.com/runs \
  -H "Content-Type: application/json" \
  -d '{
    "task": "Исследуй рынок AI-агентов за Q1 2026",
    "stream": true
  }'
```
Агент отвечает стримом событий — каждое событие это JSON-строка: прогресс, промежуточные результаты, финальный ответ.
## Почему это важно
До ACP каждый AI-проект решал проблему взаимодействия агентов по-своему: ad-hoc очереди, базы данных как шина, или агенты в одном процессе. Это работало для прототипов, но не для продакшена.
ACP даёт отрасли общий язык. Это значит:
- **Переносимость.** Агента, написанного на LangChain, можно заменить на агента на BeeAI без переписывания остальной системы
- **Композиция.** Можно собирать конвейеры из агентов разных команд и вендоров
- **Стандартизация.** Инструменты мониторинга, отладки, безопасности работают для всех агентов одинаково
- **Экосистема.** Появляются готовые ACP-агенты как сервисы — так же как REST API сервисы сегодня
Интересное время: агенты перестают быть одиночками и учатся работать в команде.
